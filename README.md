# EdgeCV

Embedded edge vision framework — real-time computer vision inference on an ESP32 camera node.

EdgeCV explores moving lightweight CV workloads off centralised servers and onto low-cost embedded hardware. The current prototype runs inference entirely on an ESP32-CAM with an OV3660 camera — no server, no cloud, no network connection required.

Two demos are included:

- **MNIST digit recogniser** — the original prototype; classifies handwritten digits from the camera feed using an INT8 CNN and the emlearn-micropython runtime
- **Custom object classifier** — the active project; a full pipeline for any two-class object recogniser, currently trained to distinguish guava vs powerbank

---

## What this achieves

A fully offline, real-time image classifier running on a microcontroller that costs under $10:

| Metric | Value |
|---|---|
| Inference time | ~184 ms per frame (~5 FPS) |
| Preprocessing (JPEG decode + convert) | ~27 ms per frame |
| Model size (int8) | ~50 KB (embedded in firmware flash) |
| Tensor arena | ~50 KB PSRAM |
| Camera | OV3660, 96×96 JPEG |
| Classification | guava vs powerbank, >90% confidence on clean samples |
| Open-world rejection | UNKNOWN output for unrecognised objects |
| Temporal stabilization | confidence averaging / EMA / vote / hysteresis |
| WiFi live stream | MJPEG to browser + JSON classification endpoint |
| Power source | USB (no external compute) |

The model runs from flash (embedded as a C array via `xxd`), loads in milliseconds, and classifies at approximately 5 frames per second. No SD card, no WiFi, no host PC needed after flashing.

## Demo 2 — Custom Object Classifier

### Pipeline overview

```
Raw images (held by hand)
        │
        ▼
sanitizer_dataset.py            removes hand (HSV mask + inpainting)
                                removes duplicate frames (perceptual hash)
        │

train_classifier_v4.py          trains TinyML scratch CNN (stronger augmentation)
                                exports int8 TFLite model
        │
        ▼
test_laptop.py                  validates model on laptop webcam before flashing
        │
        ▼
xxd -i model.tflite > main/model_data.cc    embeds model into firmware
        │
        ▼
idf.py build flash              compiles and flashes to ESP32-CAM
        │
        ▼
OV3660 → JPEG → jpg2rgb565 decode → RGB565 unpack → int8 tensor → TFLite Micro → class + confidence
```

### Model architecture (TinyML scratch CNN)

```
Input: (96, 96, 3) RGB  [Rescaling(1/255) baked in — uint8 at inference]
→ Conv2D(8,  3×3, stride=2) → ReLU → MaxPool(2×2)   output: 23×23×8
→ Conv2D(16, 3×3)           → ReLU → MaxPool(2×2)   output: 10×10×16
→ Conv2D(16, 3×3)           → ReLU                  output:  8×8×16
→ Flatten → Dropout(0.5) → Dense(16, ReLU) → Dense(1, sigmoid)

Output: single sigmoid neuron
  ≥ 0.5 → powerbank (class index 1)
  < 0.5 → guava     (class index 0)
```

Quantized to int8: ~50 KB. Fits in ESP32 flash with the default partition scheme — no SD card needed.

### Step 1 — Dataset sanitization

```bash
cd classifier
python sanitizer_dataset.py
```

Place raw images in `dataset/test_guava/` and `dataset/test_powerbank/`. The sanitizer:
- Removes hands from images using an HSV skin-tone mask + OpenCV inpainting (TELEA algorithm — no flat-colour fill)
- Detects and removes near-duplicate frames using perceptual hashing (`imagehash.phash`)
- Writes cleaned images to `dataset/guava/` and `dataset/powerbank/`

### Step 2 — Training

```bash
python train_classifier_v4.py
```

Edit the paths at the top of the file before running. The script:
- Splits data with a **group-aware 80/20 train/val split** — guarantees that original and cleaned versions of the same physical photo never appear in both train and val
- Trains with early stopping (patience=8) and LR reduction on plateau
- Evaluates per-class accuracy on the held-out test set
- Exports `model.tflite` (int8 quantized, calibrated on training data only)

### Step 3 — Laptop validation

```bash
python test_laptop.py
```

Validates the model against your laptop webcam before flashing. Edit `LABELS` at the top to match your classes.

### Step 4 — Embed model into firmware

```bash
cd classifier/firmware/main
xxd -i ../../model.tflite > model_data.cc
```

This regenerates `model_data.cc` with the new model bytes. The header `model_data.h` stays unchanged.

### Step 5 — Build and flash

```bash
cd classifier/firmware

# First time only — download ESP-IDF components
idf.py update-dependencies

# Build and flash — serial-only inference (default)
idf.py -p /dev/ttyUSB0 build flash monitor

# Build and flash — TCP stream to laptop (live_infer.py)
idf.py -DBUILD_MODE=stream -p /dev/ttyUSB0 build flash monitor

# Build and flash — WiFi + MJPEG browser stream (edit WiFi credentials first)
idf.py -DBUILD_MODE=wifi -p /dev/ttyUSB0 build flash monitor
```

The default build mode is `probe` (`main_tinyml_probe.cc`). Pass `-DBUILD_MODE=<mode>` to select
a different entry point. Valid values: `probe`, `stream`, `wifi`, `edgecv_v2`, `model_probe`, `camera_probe`.

### Step 5b — Live feed viewer (laptop)

When running `main_stream_raw.cc`, use `live_infer.py` on your laptop to
display the camera feed in real time. Inference runs on the ESP32 and prints
to serial — the laptop script does display only.

```bash
# Install dependencies (one time)
pip install opencv-python numpy

# Run the viewer
cd classifier/scripts
python live_infer.py
```

The viewer connects to `classifier-espcam.local:5555` via mDNS. On Windows,
Bonjour must be installed for `.local` resolution to work. If it fails,
hardcode the IP shown in the serial monitor:

```python
# live_infer.py — change this line
ESP32_HOST = "192.168.x.x"   # IP from serial monitor output
```

All build modes:

```bash
# probe (default) — serial-only inference, no WiFi
idf.py build flash monitor

# stream — TCP raw stream + serial inference + mDNS (laptop viewer: python live_infer.py)
idf.py -DBUILD_MODE=stream build flash monitor

# wifi — WiFi + MJPEG stream + browser UI (edit WIFI_SSID and WIFI_PASSWORD first)
idf.py -DBUILD_MODE=wifi build flash monitor

# edgecv_v2 — Phase 2: per-stage profiling + open-world rejection + temporal stabilization
idf.py -DBUILD_MODE=edgecv_v2 build flash monitor

# model_probe — confirms model loads and AllocateTensors succeeds (no camera)
idf.py -DBUILD_MODE=model_probe build flash monitor

# camera_probe — confirms camera can coexist with resident model
idf.py -DBUILD_MODE=camera_probe build flash monitor
```

The mode is passed as a CMake cache variable (`-D`), not an environment variable.
The legacy `APP_MAIN=filename.cc` env-var still works as a fallback.

### Int8 input pipeline (for reference)

The model has `layers.Rescaling(1/255)` baked into its weights. On the ESP32, do **not** divide by 255 manually. Convert camera pixels as:

```
int8_val = (int8_t)(uint8_rgb_pixel - 128)
```

This is equivalent to applying zero_point = -128. Verify by checking the printed `zero_point` value on boot — it should be -128.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| Board stuck in reset | RTS/DTR held high by USB adapter | Use `--lower-rts --lower-dtr` in picocom |
| `esp_camera_init failed: ESP_FAIL` | OV3660 doesn't support `PIXFORMAT_RGB888` | Use `PIXFORMAT_YUV422` — the firmware already does this |
| `AllocateTensors() failed` | Arena too small | Increase `TENSOR_ARENA_SIZE` in `main_tinyml_probe.cc` |
| Predictions correct but inverted | Class index order wrong | Check that folder names are exactly `guava` and `powerbank` (alphabetical = [0] guava, [1] powerbank) |
| Classifies background not object | Background bias in training data | Vary backgrounds; run sanitizer to remove hands |
| Model outputs ~50% for both classes | Training collapse or wrong input type | Check that `fill_input_tensor_int8()` is writing to `input->data.int8`, not `input->data.f` |
| Build error: `format '%d' expects int but got int32_t` | `dims->data[]` and `params.zero_point` are `int32_t` | Cast to `(int)` in log format arguments |
| `ImportError: imagehash` | Missing dependency | `pip install imagehash` |
| `cam_hal: FB-OVF` spam | `CAMERA_GRAB_LATEST` with slow consumer | Change to `CAMERA_GRAB_WHEN_EMPTY` in camera config |
| Inference produces garbage with RGB565 | Buffer sized for 3 bytes/px, loop treats packed uint16_t as raw bytes | Allocate `W*H*2` bytes; unpack uint16_t with bit shifts before int8 cast |
| Everything classified as UNKNOWN | Rejection thresholds too strict | Lower `CONFIDENCE_THRESHOLD` and/or `MARGIN_THRESHOLD` in `main_edgecv_v2.cc` |
| Prediction flips every frame at boundary | No temporal stabilization active | Enable `STABILITY_CONF_AVG` or `STABILITY_HYSTERESIS` |
| `cam_hal: NO-SOI` with JPEG pixel format | OV3660 JPEG pipeline fails at 96×96 | Use `PIXFORMAT_YUV422` + `frame2jpg()` for software encode |
| `mDNS component not found` build error | mdns not in idf_component.yml | Run `idf.py add-dependency "espressif/mdns"` |
| `socket.gaierror` on `.local` hostname | `getaddrinfo()` bypasses Windows mDNS | Use `socket.gethostbyname()` in the Python script, or hardcode the IP |
| `IPSTR`/`IP2STR` type mismatch error | Macro expects `esp_ip4_addr_t` not `struct in_addr` | Use `inet_ntoa(client_addr.sin_addr)` instead |

## Key Lessons

- The OV3660 does not support `PIXFORMAT_RGB888` — capture YUV422 and convert to RGB in firmware using BT.601 coefficients to match PIL's colour space.
- TFLite int8 models expect `int8` input, not `float32`. Writing float bytes into an int8 tensor pointer (`input->data.f` instead of `input->data.int8`) silently corrupts every pixel before inference.
- `layers.Rescaling(1/255)` inside the model means the conversion is baked into the quantized weights. Do not normalise pixel values before filling the tensor on the device.
- If a model classifies background brightness instead of object identity, vary backgrounds in training data. Pointing the camera at a white ceiling vs a dark desk should not flip the prediction.
- Group-aware train/val splitting matters when the dataset contains multiple versions of the same physical photo (e.g. original + hand-cleaned). A naive random split leaks the same scene into both sets and gives optimistic validation accuracy.
- `int32_t` is `long int` on the ESP32 toolchain. Use `(int)` casts when passing `dims->data[]` or `params.zero_point` to `%d` format specifiers, or the build fails with `-Werror=format`.
- Scratch CNNs on distinct object classes outperform MobileNetV2 transfer learning when the dataset is small and the objects are visually different. MobileNetV2 adds training time, model size, and inference latency without benefit in that regime.
- `CAMERA_GRAB_LATEST` causes constant frame buffer overflow when the consumer is slower than the DMA. Use `CAMERA_GRAB_WHEN_EMPTY` unless you specifically need the absolute latest frame.
- When changing JPEG decode functions, update the buffer size, the pixel loop, and the tensor fill together — they are coupled. RGB565 is 2 bytes/pixel packed uint16_t, not 3 separate bytes.
- A binary classifier is always closed-world — it will confidently classify a wall or keyboard as guava or powerbank. Add rejection logic before using predictions for anything real.
- Temporal stabilization is essential before driving actuators. Use hysteresis for motor control specifically: it prevents rapid alternation at decision boundaries that causes physical wear.
- The OV3660 JPEG pipeline fails at 96×96 (`NO-SOI`). Capture YUV422 and encode to JPEG with `frame2jpg()` for transmission — this also keeps inference using the same YUV frame without an extra decode step.
- `IP2STR()` and `IPSTR` macros expect `esp_ip4_addr_t*`, not `struct in_addr*`. Use `inet_ntoa()` for POSIX socket addresses.
- `mdns` is not bundled with ESP-IDF v5.3 — add it with `idf.py add-dependency "espressif/mdns"`.
- On Windows, `socket.getaddrinfo()` with explicit flags bypasses the Bonjour mDNS resolver. Use `socket.gethostbyname()` instead.
